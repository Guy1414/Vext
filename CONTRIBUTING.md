# Contributing to Vext

Thank you for your interest in contributing to Vext!

Vext started as a personal project and learning experience, and contributions from the community are welcome as the language evolves.

---

# Before you start
Please explore the internal architecture, modules, and implementation details:
https://github.com/Guy1414/Vext-Codebase-Documentation

---

## Ways to Contribute

You can help improve Vext in several ways:

- Reporting bugs
- Suggesting language features
- Improving documentation
- Optimizing compiler or VM performance
- Fixing issues
- Submitting pull requests

Before starting large changes, it is recommended to open an **Issue** first to discuss the idea.

---

## Reporting Bugs

When reporting bugs, please include:

- A clear description of the issue
- Steps to reproduce it
- Expected behavior
- Actual behavior
- Example Vext code that triggers the issue
- Your environment (OS, version of Vext, etc.)

---

## Feature Requests

Feature requests are welcome.

Please include:

- A clear description of the feature
- Why it would benefit the language
- Example syntax or usage if applicable

Language design changes should be discussed before implementation.

---

## Pull Requests

Pull requests are welcome but should follow these guidelines:

- Keep changes **focused and minimal**
- Follow the existing **code style and structure**
- Include clear commit messages
- Update documentation when necessary
- Ensure the project still builds and runs correctly

Large architectural or language changes should be discussed in an issue before submitting a PR.

---

## Code Style

Please try to match the existing style of the project.

Consistency is more important than strict formatting rules.

---

## Development Guidelines

When contributing to the compiler or VM:

- Maintain clear separation between compilation stages
  - Lexer
  - Parser
  - Semantic analysis
  - Bytecode generation
  - VM execution
- Avoid introducing unnecessary dependencies
- Prioritize clarity and performance
